# ClassCadence

> The rhythm of every great learning center.

Multi-tenant student management for supplemental learning centers. Built on Next.js 14 (App Router) + Supabase (Postgres with RLS) + Resend + Inngest, deployed to Vercel.

**Authoritative spec:** `../ClassCadence_BA_Document.docx` (sibling file in the same folder). The BA document is the source of truth — when in doubt, defer to it.

**Launch / signup plan:** `../ClassCadence_Launch_Plan.md`.

---

## Quick start

Prerequisites: Node.js 20+, pnpm, the Supabase CLI, and accounts at GitHub + Supabase + Resend + Inngest + Vercel (see launch plan Section 2).

```bash
pnpm install                          # install deps
cp .env.example .env.local            # then fill in values
supabase link --project-ref <ref>     # link to your Supabase project
supabase db push                      # apply migrations
pnpm db:types                         # generate TS types from the schema
pnpm dev                              # http://localhost:3000
```

If `supabase db push` errors on a brand-new project, open the Supabase SQL Editor and paste `supabase/migrations/0001_core_schema.sql` then `supabase/migrations/0002_rls_policies.sql` in order.

## Multi-tenant model — what you must understand before writing code

Read BA Section 11.2 first. Summary:

- **Single Postgres database.** Every tenant-owned row carries `tenant_id`.
- **Row-Level Security** (Postgres RLS) is the enforcement layer. RLS policies live in `0002_rls_policies.sql`. Three helper functions — `auth_tenant_id()`, `auth_role()`, `auth_is_super_admin()` — read the caller's profile and gate access.
- **Server code uses `createSupabaseServerClient()`**, which carries the user's JWT and is subject to RLS. Cross-tenant access is impossible by construction.
- **Service-role code uses `createSupabaseServiceClient()`**, which bypasses RLS. Use this only in:
  - Inngest scheduled jobs that legitimately span tenants (e.g., session materialization across all locations)
  - Server actions that need to write the audit log (which is RLS-protected on read but always written by the service)
  - Webhook handlers that haven't authenticated the user yet (Resend deliveries, Meta WhatsApp webhooks)

Every other code path uses the user-scoped server client. **Never put `SUPABASE_SERVICE_ROLE_KEY` in client code or in a `NEXT_PUBLIC_*` variable.**

## Build order (next steps for Claude Code)

Follow the build order in BA Launch Plan Section 8. Step 1 (start here):

> Implement Super Admin tenant creation. After this step:
> - A Super Admin (manually inserted into `user_profiles` for the first run) can log in.
> - They see a "Create tenant" form on `/admin/tenants`.
> - Submitting the form creates a row in `tenants`, a `branding_assets` row with defaults, and a Tenant Admin invite email (Resend).
> - The new Tenant Admin clicks the invite link, sets a password, and lands on `/dashboard` with their tenant context loaded.

Acceptance: `pnpm dev` and clicking through this flow works. Run `pnpm typecheck` clean. Manually verify in the Supabase dashboard that exactly one row exists in `tenants` after creation.

## Verification checklist

Before opening a PR:

- [ ] `pnpm typecheck` passes
- [ ] `pnpm build` succeeds locally
- [ ] No new `SUPABASE_SERVICE_ROLE_KEY` references in client components
- [ ] New tables (if any) have RLS enabled and a tenant-scoped policy
- [ ] Cross-tenant access test: log in as Tenant A, attempt to fetch Tenant B's data → expect empty result, never an error
- [ ] If you added a notification path, you added a `dedup_key` to prevent duplicate sends (BA FR-NT-05)
- [ ] If you added a public token-based URL, the token is HMAC-signed and time-limited (BA Security section)
